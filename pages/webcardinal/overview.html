WebCardinal and Cardinal are two OpenSource JavaScript MVC frameworks that helps the creation of the user interfaces for Self Soverign Applications (SSApps) (a concept from PrivateSky and OpenDSU). Cardinal was the first iteration and the latest one stays under the name of webcardinal (www.github.com/webcardinal) The need for a new framework arose from the negative experience with React or Angular. These frameworks through the large ecosystem and too diverse pressures of the community have become sources of inefficiency. WebCardinal and Cardinal evolve around the idea of adding a simple MVC layer over standard web components. Additionaly, controling the MVC framework offered us oportunitites to integrate OpenDSU Specific features in the controllers and therefore make it the most apopiate way of creating UIs for SSAPs. WebCardinal represents the complete rewrite version of Cardinal, with the aim to enable an even more rapid development of WEB applications. WebCardinal offers almost all the functionalities of Cardinal, either is the same form or by providing an easier alternative. From 2021, we recomand to start to migrate Cardinal SSAPPs to WebCardinal and never use the Cardinal. Most of the Web Components from Cardinal are still usable in Web Cardinal, however we encourage you to use frameworks as Ionic because their components are maintained by a larger community.

In index.html of your application (WebApp, SSApp) insert the following:

<link rel="stylesheet" href="webcardinal/webcardinal.css"> <script type="module" src="webcardinal/webcardinal.js"></script> WebCardinal offers the same configuration possibilities as Cardinal, while offering some extra ones.
Read more about it in the documentation for Configuration. WebCardinal offers some useful components to make the development of WEB applications even more easier.
Read more about each webcardinal-core component by checking the webcardinal-core package section from the WebCardinal Cheatsheet. WebCardinal comes with some very useful and powerful attributes that can be used on mainly every HTML element. data-for - improved and optimized version of the psk-for-each component data-if - improved and optimized version of the psk-condition component data-view-model - used for binding many tag attributes to the same element data-tag - used for an easier event subcription experience Considering the fact that Cardinal's psk-* components handle their own logic, flow and HTML manipulation, not all of them can integrate perfectly with the new attributes of WebCardinal. So extra care needs to be taken in this case.

WebCardinal's approach to controllers is similiar with the one from Cardinal.

Read more about Cardinal's controllers here

Currently, WebCardinal offers the following Controllers which can be extended based on the given requirements:

Controller: offers the basic functionalities of the framework WebController: offers some basic interaction with the WebCardinal components

The controlers can be used from WebCardinal.controllers available on the window object.

const { WebcController } = WebCardinal.controllers; class Page1Controller extends WebcController { initializeModel = () => ({ field1: 1, field2: 2 }); constructor(element, history) { super(element, history); this.setModel(this.initializeModel()); } } Read more about the Controller class here Read more about the WebcController class here

WebCardinal's approach to models is similiar with the one from Cardinal.

Read more about Cardinal's models here

WebCardinal went further and added a more standardised and consistent approach to models and model binding.

All of the HTML element and also custom elements have support for binding. One-way binding is ensured for every element, but there may be some limitations for two-way binding when using non-native HTML form elements. Read more about WebcCardinal's forms here.

In order to offer a consistent approach for binding, WebCardinal has support for a special attribute data-view-model.

Read more about WebCardinal's data-view-model attribute here

WebCardinal has added a new syntax that allows model binding (and also translation binding) inside the template nodes.

Message: {{ @hello }}

 const { WebcController } = WebCardinal.controllers; export default class BindingExampleController extends WebcController { initializeModel = (_) => ({ buttonText: "Button", hello: "Hello world!", }); constructor(element, history) { super(element, history); this.setModel(this.initializeModel()); } }

Message: {{ @hello }}

WebCardinal has added support for translations via skins.

Read more about WebCardinal's translation mechanism here Since WebCardinal is a complete rewrite of Cardinal, there may be some cases in which backward compatibility is not offered. One such good example of limited compatibility is with Cardinal's psk-modal, which is not at all supported in the new version. Instead, the improved webc-modal was created.

More precise, after cloning and installing webcardinal-bundler you must:

  1. Create your desired bundle npm run bundle-<BUNDLE_NAME> # example: npm run bundle-minimal
  2. Install components from bundle npm run build

The generated distribution is now in /webcardinal directory.
